# 科研 dashboard 日报说明

这份说明用于生成“科研日报”邮件。日报的目标是让 DD 快速看到统计窗口内哪些研究真的推进了、形成了什么判断、还留下了什么限制。它不是任务队列审计，也不是工具运行日志。

## 默认读取范围

- `research-dashboard/projects/*.md`
- `research-dashboard/dashboard-data.json`
- 只有 DD 后续明确要求日报包含任务队列时，才读取 `research-dashboard/tasks/requests/*.json`

不要读取、摘录、输出或暴露 token、secret、账号文件、env 文件、CodexBridge 微信账号、登录二维码、账号缓存等敏感内容。即使项目 Markdown 里维护了这些文件路径，日报也不要展开它们。

## Codex / ZCode / Chatbox 用量统计

日报自动化运行前可以先执行：

```powershell
python .\research-dashboard\tools\update_codex_usage.py
python .\research-dashboard\tools\build_dashboard.py
```

这会只读统计 `%USERPROFILE%\.codex\state_5.sqlite`、各线程 rollout JSONL、`%USERPROFILE%\.zcode` 中可解析的 ZCode 用量记录，以及 `%APPDATA%\xyz.chatboxapp.app` 中可解析的 Chatbox token 记录，更新 `research-dashboard/config/codex-usage-summary.json`，再由 dashboard 顶部第 4 个统计框按三行显示合计输入、输出、缓存命中率。Chatbox 旧记录可能只有 `tokensUsed` 总量，不能可靠拆成输入/输出；这类记录只进入 JSON 总量和 `unclassified_total_tokens`，不强行塞入 dashboard 三行。该统计用于 DD 在 dashboard 上查看系统用量，不属于科研进展；默认不要把 token 数字写进日报正文的“主要内容”或“细则”。如果 DD 明确要求日报包含用量概览，只写一短句，例如“Codex/ZCode/Chatbox 本机用量统计已更新到 dashboard 顶部”，不要展开线程标题、路径或详细数字。

## 默认发送形式

- 邮件主题固定为 `YYYY-MM-DD 科研日报`
- 正文源文本仍以纯文本保存到 `research-dashboard/daily-reports/YYYY-MM-DD.md`
- 实际发送默认走本地 SMTP，并由发送脚本把同一份正文渲染为移动端友好的 HTML 邮件
- HTML 版式以当前已确认的“科研管理”卡片样式为准：蓝色表头、右上角更新项目数、`Main Focus` 摘要块、`细则` 分段卡片
- 不在正文里额外解释“由本地 SMTP 发送”之类的链路说明

## 默认正文格式

日报正文不要写标题，不要写统计窗口，不要寒暄，也不要附加数据来源、任务队列、工具日志或文件写入说明。

正文固定写成：

```text
更新项目数：N
活跃项目数：N
主要内容：...

细则：
1：【项目一】摘要内容

2：【项目二】摘要内容

邸伟承
```

约束如下：

- `细则：` 之后每个项目都写成一整段普通文本，格式固定为 `1：【项目名】摘要内容`
- `【项目名】` 必须使用适合手机显示的短标题，不要直接使用完整论文题目或很长的 dashboard 标题；优先控制在 4 到 12 个汉字，或一个短的中英混合主题名。长题名、具体论文名和细分方向放进摘要内容里说明
- 项目编号必须按出现顺序连续递增，写成 `1：`、`2：`、`3：`，不要全部重复成 `1：`
- 项目之间空一行
- 不要使用 Markdown 表格、真正的编号列表或项目符号列表；开头的 `1：` 只是普通文本，不要触发邮箱客户端的列表样式
- 不要在项目段落内部手动换行，让邮件客户端自然折行
- 不要额外拆成“项目状态”“下一步”“待处理任务”“需要 DD 决策”“明日建议”“数据来源”等小节，除非 DD 明确要求

## 写作口径

- 语言简单直接，不用“重大突破”“显著提升”“完整闭环”等夸张词
- 重点写研究内容、形成的判断、边界条件和仍然存在的限制
- 默认不要汇报数值、参数、页数、尺寸、R2、RMSE、误差、百分比或其他细节结果；除非 DD 明确要求，否则只提纲挈领地写做了什么、形成了什么判断、还剩什么限制
- 不把工具动作当成果写。比如 `LaTeX 编译通过`、`PDF 渲染完成`、`脚本运行成功`、`页面重建完成` 默认都不写，除非它们直接改变了研究交付内容
- 如果只是稿件整理、图件整理、目录清理或交付打包，没有新增研究判断，可以直接写“完成了稿件/图件/交付整理”或“无新增研究结论”
- 如果只能从更新时间或上下文推断，而不是从项目记录里直接读到，必须标注“推断”
- 不根据文件名、路径或旧记录编造进展
- 对 pending/running/blocked 任务只做状态记录，不 claim、不改状态、不启动服务，除非 DD 明确要求处理任务

## 判断“是否纳入日报”

优先使用项目文件中统计窗口内对应日期的这些内容：

- `已解决`
- `关键记录`
- 带日期的小节
- front matter 的 `updated`

纳入日报的项目摘要应回答：

- 做了哪项研究、论文、实验、仿真或申报材料工作
- 形成了什么判断、证据或限制
- 哪些内容仍不能作为正式结论

默认不需要回答：

- 具体数值结果
- 参数表、尺寸、页数、R2、RMSE、比例或阈值
- 只适合放在论文、图注、表格或技术附录里的细节

不要只写这些内容：

- 编译通过
- 文件生成
- 页面重建
- 安装、清理、缓存、依赖处理
- agent 或工具内部过程

## 项目记录如何支持日报

后续 agent 更新 `projects/*.md` 时，`关键记录` 尽量写成可直接复用到日报的短句，例如：

```markdown
- 2026-05-26: 修正了基础力矩口径，并确认此前部分异常主要来自代码口径而不是基础物理失效；当前低阶 M2 仍不足以支撑最终论文结论，旧结果需要按失效处理后重算。
```

不建议只写这种工具过程：

```markdown
- 2026-05-26: 运行脚本、编译 PDF、渲染页面，日志无报错。
```

工具验证、编译、渲染、文件路径可以放在 `维护文件` 或最终交付说明里；日报默认不展开。
